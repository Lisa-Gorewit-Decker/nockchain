# Problem: INT8 quantization of Qwen 3.5 27B hybrid transformer

## Context

`ai-pow-vi` is a Rust crate that implements a **deterministic INT8 forward** for transformer inference. The downstream use is a verifiable-inference proof-of-work puzzle for Nockchain: two implementations of the forward must produce bit-identical outputs on the same INT8 inputs, across CPU architectures.

To validate the implementation, we compare its top-1 argmax predictions against **Ollama** (which runs the same model via llama.cpp's Q4_K_M quantization + bf16 KV-cache + fp16/bf16 intermediate activations). Target: 4/4 top-1 agreement on a 4-prompt eval set.

**Model**: Qwen 3.5 27B Vision-Language (GGUF arch tag `qwen35`)
- 64 layers: 16 standard attention every 4th layer, 48 GatedDeltaNet linear-attention hybrid layers
- IMROPE (interleaved multi-axis RoPE) on first 64 of 256 head_dim
- Hidden 5120, intermediate 17408
- Std attention: 24 Q heads, 4 KV heads, head_dim 256, packed Q+gate
- Hybrid (GatedDeltaNet): 16 K heads, 48 V heads, head_k = head_v = 128, conv kernel = 4, conv channels = 10240
- ~27B parameters total

## What we've validated

1. **Architecture is correct.** A pure-f32 reference forward (in `crates/ai-pow-vi/src/bin/calibrate.rs`) achieves **4/4 top-1 vs Ollama** on the eval set. So the operator sequence (attention, IMROPE, DeltaNet recurrence, gated RMSNorm, residual paths, etc.) faithfully reproduces llama.cpp's `qwen35` graph at `src/models/qwen35.cpp`.

2. **Per-tensor weight scales are plumbed.** Every i8 weight tensor's `max(|w|)` is captured at convert time. For tensors with `max(|w|)` materially different from 1 (notably `ssm_a` with `max ≈ 16`), this scale is stored in the manifest and applied at runtime dequant.

3. **Combined matmul scales.** The matmul rescale formula
   ```
   out_i8 = round(acc × scale.num / 2^15)
   ```
   is mathematically correct iff `scale.num = (a_max × w_max / (127 × out_max)) × 2^15`. The current converter computes this combined scale at convert time:
   ```
   combined.num = (a_scale.num × w_scale.num) / out_scale.num
   ```

4. **Per-tap activation calibration data exists.** A 1539-entry `scales.json` records `max(|x|)` per quantization tap, generated by the f32 reference on the eval prompts.

## What's broken — saturation

When we instrument the i8 forward to record per-layer output max-abs, we see this pattern:

| Layer | Max |x_i8| (averaged across 4 prompts) |
|---|---|
| 0 | 109 |
| 1 | 111 |
| 2 | 115 |
| 3+ | **128** (saturated at i8::MIN = -128, i8::MAX = 127) |

**Every layer from layer 3 onwards is fully saturated** in the i8 representation. Once saturated, the residual add at the end of each layer (in i8 with saturating arithmetic) loses all signal — outputs become essentially `±128` sign noise.

We've tried tightening all output scales by factors of 4×, 8×, 16×, 32× (which effectively divides the runtime combined-scale numerator by the same factor, making i8 outputs proportionally smaller). At every level, post-rescale i8 still saturates at 128 from layer 3 onward.

The math: a typical matmul accumulator with k=5120 reduction has magnitude ~1M (sum of i8×i8 products). Even with scale × 32 (combined.num ≈ 81), `acc × 81 / 32768 ≈ 2700`, which clamps to i8::MAX. To get post-rescale below 100, we'd need scales ~3, which is below the 1/32768 representable minimum.

## Constraints — what we can't change

This crate's design contract:

- **Per-tensor symmetric INT8** (single scale per tensor, zero-point = 0, no per-channel, no per-group).
- **Scale stored as `i32 num / 2^15`** fixed-point (positive, denominator locked).
- **i32 accumulators** for matmul (k ≤ 2^15 to avoid overflow).
- **Banker's rounding (round-half-to-even)** on every rescale, `#[inline(never)]` to prevent the compiler from fusing it.
- **Reduction order is row-major, ascending index** — vendor SIMD kernels that reorder reductions are non-conformant by construction.
- **Residual adds happen in i8 with saturating arithmetic** (this is what's saturating, but changing it requires a wire-format / API change that breaks the determinism contract with existing implementations).

These constraints make ai-pow-vi a "naive symmetric INT8" reference, which is known to be lossy on transformer activations. The question is whether there's a calibration / pre-processing strategy compatible with these constraints that can still match Ollama (which uses richer Q4_K_M + bf16 routing).

## Specific questions for research

1. **Can per-tensor symmetric INT8 with i8 residual adds match bf16-grade transformer inference?** What does the literature say about top-1 fidelity at this constraint level on 27B-class models?

2. **Outlier handling in symmetric quantization.** Transformer activations have heavy-tailed distributions — a few channels carry the outliers, and they dominate the per-tensor max. Approaches like SmoothQuant migrate outliers to weights (which are easier to quantize). Can SmoothQuant-style transformations be applied **at convert time only** (no runtime change) compatible with our per-tensor symmetric constraint?

3. **Activation calibration.** Our scales come from a fp32 reference forward. Is there a known technique to calibrate scales *for the i8 path*, accounting for the saturation/quantization noise the i8 path actually produces? Specifically:
   - Self-calibration: iterate i8 forward → record actual i8 mags → adjust scales → re-converge.
   - Knowledge-distillation-style: pick scales that minimize KL between i8 output logits and fp32 logits.

4. **Layer-by-layer accumulating error.** With 64 layers each introducing some quantization noise, even small per-layer error compounds. Is there a tractable approach (mixed precision per layer? higher-precision residual stream?) compatible with our determinism contract?

5. **Hybrid architecture-specific.** Qwen 3.5 27B has 48 GatedDeltaNet (linear-attention) layers with their own recurrence. Are there published quantization results for linear-attention models? They behave differently than softmax attention.

6. **Practical workarounds.** Even if matching Ollama at 4/4 isn't feasible under these constraints, are there known "good enough" thresholds for proof-of-work-style use cases where bit-identical determinism is the primary goal and absolute fidelity to bf16 inference is secondary?

7. **Specific to llama.cpp's Q4_K_M reference.** What's the right reference output to compare against? Q4_K_M has its own quantization noise relative to fp16. If we recalibrated against Q4_K_M's *actual* outputs (rather than our fp32-of-Q4_K-weights forward), would that change anything? I.e., is our "fp32 forward over dequantized Q4_K weights" itself the right oracle?
